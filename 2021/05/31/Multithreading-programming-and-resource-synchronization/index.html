Basic concepts of threads

A process represents a program actually running on a computer. Each process has its own independent process address space and context stack. However, for the operation performed by a program itself, the process actually does nothing (does not execute any process code), only provides a large environment container, and the actual execution body in the process is a thread. Therefore, there must be at least one thread in a process. We call this thread the main thread.

  • [program]: The ordered combination of a set of instructions， which is a static instruction and exists forever.
  • [Process]: It is an independent unit for system resource allocation and scheduling.
  • [Thread]: It is an entity of a process and a basic unit of CPU scheduling and allocation.

Process V.S. Thread

  • A thread must belong to a process, and a process can have multiple threads and at least one thread.
  • All threads belonging to a process share all resources of the process.
  • A process is an execution of a program, and a thread is the execution of a program fragment in a program.
  • Each process has its own memory space, and threads share the memory space of the process they belong to.

1. Linux

1.1 Mutex

The usage of Linux mutex is similar to that of windows critical section object. Generally, resources are protected by limiting multiple threads to execute a piece of code at the same time.

We can see that after adding this mutex, the second thread will not start executing until the first thread has performed 10 operations, so the resource number is protected.
Then, Let’s talk about Mutex Lock’s attribute type:

  • PTHREAD_MUTEX_NORMAL This is the default property of the mutex object. After a resource is locked with the PTHREAD_MUTEX_NORMAL, other threads will be blocked from calling a until the resource is released.
  • PTHREAD_MUTEX_ERRORCHECK If the property of the mutex is PTHREAD_MUTEX_ ERROCHECK, the current thread repeatedly calls pthread_ mutex_ lock will directly return EDEADLOCK. If other threads call again on this mutex_ mutex_ lock, it will be blocked at the call of the function.
  • PTHREAD_MUTEX_RECURSIVE This property allows the same thread to re lock the mutex it holds.

1.2 Signal

In order to prevent a series of problems caused by multiple programs accessing a shared resource at the same time, we need a method that can authorize by generating and using tokens. At any time, only one execution thread can access the critical area of the code. Critical areas are those areas where code that performs data updates needs to be executed exclusively. Semaphores can provide such an access mechanism that only one thread in a critical area is accessing it at the same time, that is, semaphores are used to coordinate the process’s access to shared resources.

1.4 Read/Write Lock

  • Read Lock Allow multiple threads to read resources at the same time.
  • Write Lock If the current resource is occupied by a write lock, access to other processes will be blocked.
  • When a read-write lock is used, reading and writing are not allowed at the same time. Reading and writing are mutually exclusive.

This is the simplest and easiest to use. It is used to prevent multiple threads from executing their protected code segments at the same time. In other words, the code in the critical section area allows only one thread to execute at a certain time.

In programming, atomic operation means that it takes effect in one occurrence. An atomic operation cannot be stopped halfway: it either occurs completely or does not occur. Atomic operations without side effects are visible until completion.

ThreadPools and TaskQueue

Thread pool is actually just a group of threads. In general, we need to execute some tasks asynchronously. The generation and execution of these tasks exist in the whole life cycle of the program. Instead of letting the operating system frequently create and destroy threads for us, it is better to create a set of threads that will not exit during the program life cycle. In order not to waste system resources, our basic requirement is that when a task needs to be executed, these threads can automatically get the task and execute it. When there are no tasks, these threads are blocked or asleep.

Since many tasks will be generated in the program life cycle, these tasks must have a storage place, which is the queue. There are many methods for queue design, such as putting tasks from the tail and taking them out from the head, or putting tasks from the head and taking them out from the tail, and even adding priorities.

Therefore, it is obvious that this is essentially the producer consumer model. The producer is the scene of generating tasks, and the threads in the thread pool are consumers. Of course, this is not absolute.

Since multiple threads will operate the queue at the same time, according to the principle of multithreaded programs, we generally need to lock the queue to avoid unexpected results caused by multithreading competition.
